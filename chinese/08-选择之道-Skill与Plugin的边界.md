# 番外篇：选择之道——Skill 与 Plugin 的边界

> "Skill 是轻量级的能力封装，Plugin 是重量级的功能打包"
>
> "Skill 适合快速迭代和个人使用，Plugin 适合团队标准化和长期维护"
>
> "它们不是竞争关系，而是不同层级的解决方案"

## 一个困惑的诞生

在本书中，我们一直在讨论 Skills——它的本质、设计方法、工程实践。但当我第一次深入使用 Claude Code 时，我发现了一个让我困惑的现象。

当我按下斜杠键（/）调出命令列表时，我看到一长串可以调用的命令。有些是我自己写的 Skills，有些是我安装的 Plugins。但在那个列表里，它们看起来是一样的——都是一个名字、一段描述、一个可以点击的选项。

我分不清哪个是 Skill，哪个是 Plugin。

这个困惑引发了我的思考：既然它们在用户界面中长得一样，都能用斜杠命令调用，那为什么要有两个概念？它们到底有什么差异？还是说这只是产品设计的混乱？

经过一段时间的探索和使用，我逐渐理解了：Skill 和 Plugin 确实是两种不同的东西，但它们在用户界面上的"模糊"是有意为之的设计。它们解决的是不同层次的问题，服务于不同的使用场景。

这个番外篇，就是要把这个理解过程分享给你——从"分不清"到"明白差异"，再到"知道什么时候用什么"。

## 官方定义：Standalone vs Plugins

根据 Claude Code 官方文档的定义，有两种方式来添加自定义功能：

| 方式 | 技能名称 | 最适合 |
|------|---------|--------|
| **Standalone**（`.claude/` 目录） | `/hello` | 个人工作流、项目特定定制、快速实验 |
| **Plugins**（带 `.claude-plugin/plugin.json` 的目录） | `/plugin-name:hello` | 团队分享、社区分发、版本发布、跨项目复用 |

**什么时候用 Standalone（我们说的 Skill）：**
- 你只为单个项目定制 Claude Code
- 配置是私人的，不需要分享
- 你在打包前先实验 skills 或 hooks
- 你想要简短的技能名称，如 `/hello` 或 `/deploy`

**什么时候用 Plugins：**
- 你想与团队或社区分享功能
- 你需要在多个项目中使用相同的 skills/agents
- 你想要扩展的版本控制和简单更新
- 你通过 marketplace 分发
- 你可以接受带命名空间的技能名称，如 `/my-plugin:hello`（命名空间防止插件间冲突）

官方建议是：**先在 `.claude/` 中用 standalone 配置快速迭代，准备分享时再转换为 plugin。**

## 核心定位：能力封装 vs 功能打包

基于官方定义，我们可以更清晰地概括两者的区别：

- **Skill（Standalone）= 轻量级的能力封装**
- **Plugin = 重量级的功能打包**

Skill 是对某种特定能力的封装——比如"翻译这段文字"、"整理这些文件"、"写这个函数"。它聚焦于单一功能，力求简单、快速、灵活。存放在项目的 `.claude/` 目录中，技能名称简短（如 `/hello`），适合个人使用和快速实验。

Plugin 是对多种功能的打包——比如"一整套前端开发工作流"、"团队协作用具包"、"企业级 CI/CD 流程"。它整合了多个 Skills、多个 agents、多个 hooks、多个 MCP 配置，形成一个完整的解决方案。有独立的目录结构，技能名称带命名空间前缀（如 `/my-plugin:hello`），适合团队分享和社区分发。

## 多维度对比

让我们从多个维度来对比 Skill 和 Plugin。

| 对比维度 | Skill（Standalone） | Plugin |
|---------|-------------------|--------|
| **复杂度** | 简单，通常只有一个 SKILL.md | 复杂，有严格目录结构和多个组件 |
| **命名方式** | 简短名称如 `/hello` | 命名空间如 `/my-plugin:hello` |
| **适用规模** | 小规模、单一功能 | 大规模、多功能整合 |
| **分享方式** | 单独分享文件或目录 | 整体打包，通过 marketplace 分发 |
| **版本管理** | 通常无正式版本管理 | 语义化版本（Semantic Versioning） |
| **维护成本** | 低，快速迭代 | 高，需谨慎测试 |
| **使用门槛** | 低，个人和小团队 | 高，需学习成本 |
| **热部署** | ✅ 支持实时变化检测 | ❌ 需手动 `/reload-plugins` |
| **目录位置** | 项目的 `.claude/skills/` | 专门的 Plugin 目录 |
| **包含内容** | SKILL.md + 可选的 scripts/templates/references | skills/agents/commands/hooks/MCP/LSP/settings |

**复杂度：**

Skill 是简单的。一个 Skill 通常只有一个 SKILL.md 文件，可能加上一些 scripts 和 references。存放在 `.claude/skills/` 目录中。你可以在几分钟内创建一个 Skill，在几小时内完善它。

Plugin 是复杂的。一个 Plugin 有严格的目录结构：`.claude-plugin/plugin.json`（清单文件）、`skills/`（Agent Skills）、`agents/`（自定义 agents）、`hooks/`（事件处理器）、可能还有 `.mcp.json`（MCP 配置）、`.lsp.json`（LSP 配置）、`settings.json`（默认设置）。创建一个 Plugin 通常需要几天甚至几周。

**命名方式：**

Skill 使用简短名称。调用时如 `/hello`、`/deploy`、`/review`。名称是全局的，不同项目的 Skill 可能重名（但在各自项目中独立）。

Plugin 使用命名空间名称。调用时如 `/my-plugin:hello`、`/frontend-toolkit:deploy`。命名空间前缀防止了不同 Plugin 之间的命名冲突。

**适用规模：**

Skill 适合小规模、单一功能的场景。如果你只需要解决一个特定问题——比如"帮我检查代码风格"——Skill 是最佳选择。

Plugin 适合大规模、多功能的场景。如果你需要整合一整套工具链——比如"前端开发从代码检查到构建部署的全流程"——Plugin 是更好的选择。

**分享方式：**

Skill 可以单独分享。你可以把 `.claude/` 目录发给朋友，或者把 SKILL.md 文件贴给别人。使用者只需要把文件放到正确的目录就能使用。

Plugin 必须整体打包。通过 marketplace 分发，使用者安装整个 Plugin。Plugin 内部的各个部分不能单独安装。

**版本管理：**

Skill 通常没有正式的版本管理。你可能用文件名或文件夹名来区分版本（如 `v1`、`v2`），但这不是标准做法。

Plugin 使用语义化版本（Semantic Versioning）。在 `plugin.json` 中定义 `version` 字段（如 `1.0.0`），支持主版本、次版本、修订版本的升级规则。这让 Plugin 的更新更加规范。

**维护成本：**

Skill 的维护成本很低。因为它功能单一、结构简单，修改和更新都很方便。你可以快速迭代，频繁发布新版本。

Plugin 的维护成本较高。因为它功能复杂、依赖众多，每次修改都需要考虑对其他部分的影响。你需要更谨慎地测试，更规范地管理版本。

**使用门槛：**

Skill 的使用门槛很低。用户只需要知道 Skill 的名字和基本用法，就可以直接调用。适合个人开发者和小团队。

Plugin 的使用门槛较高。用户需要理解 Plugin 包含哪些功能、如何配置、如何组合使用。这通常需要一定的学习成本。适合中大型团队和企业级部署。

**热部署支持：**

这是一个重要的差异，经常被忽视。

**Skills 支持热部署。** 通过 `--add-dir` 添加的目录中的 Skills 支持**实时变化检测（live change detection）**。你可以在会话中编辑 SKILL.md 文件，修改会自动被检测并应用，无需重启 Claude Code。这对于开发和调试来说非常方便——改一点，测一点，不需要反复重启。

**Plugins 需要手动重载。** 修改 Plugin 后，你需要手动运行 `/reload-plugins` 命令来重新加载。系统不会自动检测 Plugin 的变化。这个设计可能是为了性能考虑——Plugin 通常更复杂，频繁检测变化可能带来开销。

这意味着如果你在开发过程中频繁修改，Skills 的热部署特性会显著提升你的开发效率。而 Plugins 更适合"稳定发布"的场景——安装后很少修改，修改后通过 `/reload-plugins` 手动重载。

**目录位置：**

Skill 存放在项目的 `.claude/skills/` 目录中。每个项目有自己独立的 Skills，互不干扰。

Plugin 存放在专门的 Plugin 目录中，通过 marketplace 或本地路径安装。安装后在所有项目中可用（除非被项目设置禁用）。

**包含内容：**

Skill 主要包含 SKILL.md（必需），可选的 scripts/、templates/、references/、config.json。

Plugin 可以包含：skills/（Agent Skills）、agents/（自定义 agents）、commands/（Markdown 命令）、hooks/（事件处理器）、.mcp.json（MCP 配置）、.lsp.json（LSP 配置）、settings.json（默认设置）。

## 什么时候用 Skill？

Skill 是默认选择。如果你不确定用什么，先用 Skill。

**适合用 Skill 的场景：**

你需要快速实现一个功能。从想法到可用，Skill 可以在几小时内完成。

你的功能需求相对简单。不需要复杂的配置，不需要多个工具协作。

你在个人项目或小团队中使用。不需要严格的标准化，可以容忍一些差异。

你希望频繁迭代。快速响应用户反馈，不断优化功能。

你在探索某个想法。还不确定最终形态，需要边做边调整。

你想要简短的技能名称。`/hello` 比 `/my-plugin:hello` 更容易输入和记忆。

**不适合用 Skill 的场景：**

你需要整合多个功能，并且希望它们作为一个整体被分享和部署。

你的团队需要严格的标准化配置。每个人都必须使用完全相同的工具链。

你需要版本控制和规范的更新流程。

你需要通过 marketplace 分发你的解决方案。

## 什么时候用 Plugin？

Plugin 是进阶选择。当你发现 Skills 无法满足需求时，再考虑 Plugin。

**适合用 Plugin 的场景：**

你需要整合一整套工具链。比如"前端开发工作流"包含代码检查、格式化、测试、构建、部署等多个环节。

你的团队需要标准化配置。比如所有成员使用相同的代码规范、相同的文档模板、相同的工作流程。

你的功能需要复杂的配置。比如需要配置多个 MCP、需要设置各种密钥和权限、需要处理复杂的依赖关系。

你希望"一键安装，开箱即用"。用户不需要逐个配置 Skills，只需要安装 Plugin 就能获得全部功能。

你在构建可分发的解决方案。比如你要为某个行业、某个团队、某个特定用途提供一整套工具。

你需要通过 marketplace 分发。官方 marketplace 让 Plugin 的发现和安装变得简单。

你需要命名空间来避免冲突。当你的功能与其他人开发的功能重名时，命名空间前缀可以防止冲突。

**不适合用 Plugin 的场景：**

你只是需要一个简单的功能。Plugin 对你来说是大材小用。

你希望频繁修改和迭代。Plugin 的维护成本会让你不堪重负。

你的用户技术背景各异。Plugin 的使用门槛可能会挡住一部分用户。

## 迁移路径：从 Skill 到 Plugin

官方文档明确建议了一条迁移路径：**先在 `.claude/` 中用 standalone 配置快速迭代，准备分享时再转换为 plugin。**

这是一个非常实用的建议。很多功能一开始不需要 Plugin 的复杂性，用 Skill 快速验证可行性。当功能成熟、有分享需求、需要标准化时，再转换为 Plugin。

转换过程包括：创建 Plugin 目录结构、添加 `plugin.json` 清单文件、将 Skills 移动到 `skills/` 目录、添加命名空间前缀、测试和发布。

## 它们是互补的，不是竞争的

最后，我想强调一点：Skill 和 Plugin 不是竞争关系，而是互补关系。

**一个 Plugin 可以包含多个 Skills。** 这是最常见的组合方式。Plugin 的 `skills/` 目录中存放多个 Skills，它们共享 Plugin 的配置和命名空间。

**你可以先用 Skill 验证想法，再用 Plugin 打包成熟方案。** 这是很多开发者的实际路径：先做一个 Skill 快速验证可行性，如果成功了，再把它整合到一个 Plugin 中，作为标准化方案分享给团队或社区。

**你可以同时使用 Skills 和 Plugins。** 你可以安装一些通用的 Plugins 来获得基础能力，也可以在项目的 `.claude/` 目录中创建自己的 Skills 来满足个性化需求。它们不会互相冲突，而是互相补充。

## 选择建议

如果你的团队在考虑用 Skill 还是 Plugin，我的建议是：

**对于个人开发者和小团队：** 从 Skills 开始。它们简单、灵活、够用。等你的需求变得复杂、团队规模扩大、需要标准化时，再考虑迁移到 Plugin。

**对于中大型团队：** 可以考虑直接用 Plugin。标准化带来的收益会超过维护成本。但即便如此，你也可以在 Plugin 内部使用 Skills 来实现具体功能，保持模块化和灵活性。

**对于开源项目：** 提供一个 Plugin 是很好的选择。用户可以一键安装，获得完整体验。官方 marketplace 让发现和安装变得简单。

**对于产品团队：** Plugin 是更好的选择。你可以把一整套工作流打包成 Plugin，作为产品的一部分交付给客户。这比要求客户逐个配置 Skills 要专业得多。

**命名建议：** 个人项目用简短的 Skill 名称（`/hello`），团队和社区项目用命名空间的 Plugin 名称（`/my-plugin:hello`）。这样既能保持个人开发的便利性，又能避免团队协作时的冲突。

## 写在最后

Skill 和 Plugin 的区别，本质上是"工具"和"工具箱"的区别。

当你需要一把螺丝刀时，你只需要一个 Skill。当你需要一整套工具箱时，你需要一个 Plugin。

不要被选择困扰。从 Skill 开始，需要时再升级到 Plugin。这是大多数情况下的最佳路径。
