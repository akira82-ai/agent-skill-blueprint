## 从单文件到系统：目录结构的认知跃迁

在前三章中，我们讨论了 Skill 的本质、产品化思维和参数化设计。这些都是"怎么想"的问题——如何理解 Skill、如何用产品化的方式思考它、如何设计它的接口。现在我们来讨论"怎么做"的问题——如何组织一个 Skill 的内部结构。

大多数人对 Skills 的认知停留在"写一个 SKILL.md 就够了"的层面。这个认知不是错的，但它只发挥了 Skill 能力的一小部分。一个完整的 Skill 不仅仅是 prompt 封装，它是一个有内部结构、有专业分工、有扩展能力的"微型系统"。

为什么会有这样的认知偏差？因为 SKILL.md 确实是 Skill 的唯一必需部分。你可以只写一个 SKILL.md，就能让 Skill 正常工作。这种"最小可用"的特性，让很多人误以为这就是 Skill 的全部形态。但事实上，SKILL.md 只是冰山一角。

```
┌─────────────────────────────────────────────────┐
│              Skill 能力冰山图                     │
├─────────────────────────────────────────────────┤
│                                                 │
│   ┌─────────────────────────────────────┐       │
│   │   水面以上：SKILL.md (必需)          │       │
```

只使用 SKILL.md 的人，就像买了一台智能手机却只用来打电话——能用，但你错过了 90% 的功能。

## 完整的目录结构

一个标准的 Skill 目录包含以下内容：

```
skill-name/
├── SKILL.md          # 主文件（必需）
├── scripts/          # 可执行脚本
├── templates/        # 模板文件
├── references/       # 知识库
└── config.json       # 配置文件（可选）
```

这个结构看似简单，但每个部分都有其特定的价值。为了更清晰地理解，这里有一个详细的对照表：

| 目录/文件 | 是否必需 | 核心职责 | 存放内容 | 使用场景 |
|----------|---------|---------|---------|---------|
| **SKILL.md** | ✅ 必需 | 思考与决策 | frontmatter + 指令内容 | 所有 Skill |
| **scripts/** | ❌ 可选 | 执行与实现 | Python/JS/Shell 脚本 | 需要精确执行的任务 |
| **templates/** | ❌ 可选 | 格式标准化 | 代码/文档/邮件模板 | 需要格式化输出的 Skill |
| **references/** | ❌ 可选 | 知识沉淀 | 行业文档/案例/术语表 | 需要专业知识的 Skill |
| **config.json** | ❌ 可选 | 配置管理 | API密钥/路径/偏好设置 | 需要预配置的 Skill |

很多人只用 SKILL.md，完全忽略了其他部分。这就像买了一台电脑，却只用来打字——它能用，但你没有发挥它十分之一的能力。

### 真实案例：file-organizer 的 SKILL.md

为了让这个目录结构更具体，这里有一个来自 GitHub 仓库 `akira82-ai/file-organizer` 的真实 SKILL.md 示例（节选 frontmatter 部分）：

```yaml
---
name: file-organizer
description: |
  基于 AI 语义分析的智能文件整理技能，根据文件内容动态创建 Johnny Decimal 分类。

  当用户提到以下内容时触发：
  - "整理文件"、"分类文件"、"组织下载"、"文件归档"
  - "文件太多了"、"下载文件夹很乱"、"文件乱七八糟"
  - "把文件按类别整理"、"自动分类文件"
  - "清理下载文件夹"、"整理文档"
  - "文件管理"、"文件分类整理"
  - 提到需要将文件从一个目录移动到另一个目录进行分类

  核心功能：
  - AI 分析文件名语义，动态生成分类
  - 使用 Johnny Decimal 编号系统（XX.YY 格式）
  - 自动检测重复文件（MD5 算法）
  - 用户确认分类计划后再执行
version: 1.0.0
allowed-tools: Bash, AskUserQuestion, Task
author: Claude Code
---
```

这个 frontmatter 结构展示了 SKILL.md 的核心要素：
- **name**: Skill 的唯一标识符
- **description**: 详细的功能描述和触发条件（这是最重要的部分！）
- **version**: 语义化版本号
- **allowed-tools**: 声明这个 Skill 需要使用哪些工具
- **author**: 作者信息，建立信任和联系方式

## scripts 目录：从智能到能力

在所有被低估的部分中，scripts 目录可能是最被低估的一个。

**scripts 的核心价值是什么？** 它让 Skill 不仅能"思考"，还能"执行"。

SKILL.md 中的 prompt 告诉 Claude "怎么思考"——理解用户意图、分析问题、制定方案。但有些事情不适合让大模型来做。为了更清晰地划分职责，这里有一个对比：

| 任务类型 | 大模型擅长 | 代码擅长 | 选择建议 |
|---------|-----------|---------|---------|
| **理解意图** | ✅ 强项 | ❌ 困难 | 用 SKILL.md |
| **制定方案** | ✅ 强项 | ❌ 困难 | 用 SKILL.md |
| **简单计算** | ⚠️ 可以 | ✅ 更好 | 简单用 SKILL.md，复杂用 scripts |
| **批量文件操作** | ❌ 不稳定 | ✅ 强项 | 用 scripts |
| **精确逻辑控制** | ❌ 易出错 | ✅ 强项 | 用 scripts |
| **API 调用** | ⚠️ 可以 | ✅ 更可靠 | 用 scripts |
| **创意写作** | ✅ 强项 | ❌ 困难 | 用 SKILL.md |
| **数据挖掘** | ❌ 有限 | ✅ 强项 | 用 scripts |

这些任务的特点是：复杂但稳定。说它们复杂，是因为实现起来需要很多代码；说它们稳定，是因为逻辑不会因为场景不同而改变。对于这类任务，我更倾向于用传统的代码来实现，而不是让大模型去"推理"。

这就是 scripts 目录存在的意义：**把复杂但稳定的处理从大模型中剥离出来，交给传统代码去执行。**

这种分工带来了巨大的好处。大模型负责它擅长的——理解、推理、创造、沟通；代码负责它擅长的——精确、高效、稳定、可靠。两者各司其职，Skill 也就从一个"聪明的助手"变成了"聪明的助手 + 能干的执行者"。

## scripts 的强大能力

为什么说 scripts 被低估了？因为它的能力远超大多数人的想象。让我用表格来展示 scripts 能够做什么：

| 能力类型 | 说明 | 实际应用示例 |
|---------|------|------------|
| **本地应用执行** | 运行电脑上的任何程序 | 调用 ImageMagick 处理图片、调用 ffmpeg 转换视频 |
| **高权限访问** | 访问网络、读写文件、调用 API | 批量下载文件、修改系统配置、调用云服务 API |
| **第三方库调用** | 继承整个开源生态 | 用 pandas 处理数据、用 PIL 识别图像、用 scikit-learn 跑模型 |
| **复杂运算处理** | 精确的数学和逻辑运算 | 统计分析、数据挖掘、机器学习推理 |
| **系统集成** | 与系统级功能交互 | 操作系统命令、数据库操作、消息队列 |

**本地应用执行能力。** scripts 可以运行你电脑上的任何程序。你想在 Skill 中调用图片处理工具？可以。你想调用系统命令？可以。你想运行自己写的任何可执行文件？都可以。这让 Skill 的能力边界被大大扩展了。

**高权限访问能力。** 相比于普通的对话，scripts 拥有更高的权限——它可以访问网络、读取文件、写入文件、调用系统 API。这些能力让 Skill 能够完成普通 prompt 无法完成的任务。

**第三方库调用能力。** 如果你是 Python 开发者，你可以用 pip 安装任何你需要的库；如果你用 Node.js，你可以用 npm 安装任何包。这意味着 Skill 可以直接继承整个开源生态的能力——数据处理、机器学习、图像识别、自然语言处理，任何你能想到的功能，都可能被集成到 Skill 中。

**复杂运算处理能力。** 大模型不是计算器，它不擅长精确的数学运算。但 scripts 可以——无论是统计运算、数据挖掘，还是跑机器学习模型，都可以在 scripts 中精确完成。

### 真实案例：insights-zh 的 scripts 实现

为了更具体地展示 scripts 的能力，这里有一个来自 `insights-zh` 的真实代码示例。这个 skill 需要解析 HTML 报告、提取文本块、管理翻译进度、批量更新翻译结果——这些都是复杂但稳定的任务，非常适合用 scripts 实现。

以下是其 `translate.py` 脚本的核心代码（节选自 GitHub `akira82-ai/insights-zh`）：

```python
#!/usr/bin/env python3
"""
Insights 报告翻译脚本（重构版）
使用 TSV 格式 + Task 子代理，避免嵌套会话和 JSON 转义问题
"""
import os
import sys
import csv
import subprocess
from pathlib import Path
from datetime import datetime, timedelta
from bs4 import BeautifulSoup

# ==================== 配置 ====================
DEFAULT_REPORT_PATH = Path.home() / '.claude/usage-data/report.html'
MAX_REPORT_AGE_DAYS = 3


def check_report_freshness(report_path, max_days=MAX_REPORT_AGE_DAYS):
    """检查报告是否新鲜（max_days 天内）

    Returns:
        tuple: (is_fresh, message)
            - is_fresh: None=文件不存在, True=新鲜, False=过期
            - message: 描述信息
    """
    report_path = Path(report_path)

    if not report_path.exists():
        return None, f"报告文件不存在: {report_path}"

    mtime = report_path.stat().st_mtime
    file_date = datetime.fromtimestamp(mtime)
    age_days = (datetime.now() - file_date).days

    if age_days > max_days:
        return False, f"报告已过期 {age_days - max_days} 天"

    return True, f"报告新鲜（{age_days} 天前生成）"


def extract_text_blocks(html_path):
    """使用 BeautifulSoup 提取文本块"""
    with open(html_path, 'r', encoding='utf-8') as f:
        html_content = f.read()

    soup = BeautifulSoup(html_content, 'html.parser')
    nodes = []

    for string in soup.find_all(string=True):
        # 跳过 script/style 和文档节点
        if string.parent.name in ['script', 'style', '[document]']:
            continue

        text = string.strip()
        if text:
            nodes.append((string, text))

    return nodes


def save_to_tsv(nodes, tsv_path):
    """保存文本块到 TSV 文件"""
    with open(tsv_path, 'w', encoding='utf-8') as f:
        writer = csv.writer(f, delimiter='\t')
        for node, text in nodes:
            writer.writerow([text, ''])  # 原文、译文（空）
```

这个代码片段展示了 scripts 的几个核心能力：

**HTML 解析能力。** 使用 `BeautifulSoup` 库解析 HTML 报告，提取所有文本节点。这涉及到复杂的 DOM 遍历和节点过滤，如果用大模型来做，结果会不稳定且难以预测。

**文件系统操作。** 检查文件是否存在、读取文件修改时间、计算文件年龄。这些都是需要精确结果的操作，不适合让大模型"猜测"。

**数据结构处理。** 使用 TSV 格式存储原文和译文，确保数据结构的完整性和一致性。这涉及到 CSV 读写、数据验证、格式转换等复杂逻辑。

**错误处理。** 函数返回明确的状态码（None/True/False）和描述信息，让调用方能够根据不同情况做出不同决策。这种精确的错误处理是代码的优势。

同样的，`auto_batch.py` 展示了批量处理和进度管理的实现：

```python
#!/usr/bin/env python3
"""
自动化批量翻译脚本
在主会话中逐批翻译，确保 100% 完成
"""
import sys
import csv
from pathlib import Path

TSV_PATH = Path('/tmp/insights_blocks.tsv')
BATCH_SIZE = 15


def print_progress(rows):
    """打印进度条"""
    total = len(rows)
    translated = sum(1 for row in rows if len(row) >= 2 and row[1].strip())
    percent = translated / total * 100 if total > 0 else 0

    bar_length = 40
    filled = int(bar_length * translated / total)
    bar = '█' * filled + '░' * (bar_length - filled)

    print(f"\n进度: [{bar}] {percent:.1f}% ({translated}/{total})")
    return translated == total


def get_next_batch_indices(rows, batch_size):
    """获取下一批未翻译的行索引"""
    indices = []
    for i, row in enumerate(rows):
        if len(row) < 2 or not row[1].strip():
            indices.append(i)
            if len(indices) >= batch_size:
                break
    return indices
```

这段代码展示了用户体验设计的实现：实时进度条、批量处理、状态跟踪。这些都是用纯代码实现的，保证了执行的稳定性和进度显示的准确性。

这些能力的代价是什么？是**安全风险**。

正因为 scripts 可以做这么多事情，它的权限也非常高。如果一个 Skill 来自你不信任的来源，它的 scripts 可能会做坏事——窃取你的文件、监控你的操作、上传你的数据。这不是危言耸听，而是目前 Skill 生态已经暴露出的真实问题。

前段时间就有一个案例：Claude Code 使用的某个第三方库被发现被植入了渗透性代码。这个库本身是正常的，但某个版本被恶意篡改了，所有更新到这个版本的用户都中招了。这件事给我们敲响了警钟：**高权限带来强大能力，也带来安全风险，这是必须支付的代价。**

作为用户，你需要谨慎选择 Skill 的来源；作为开发者，你需要对安全性负责。这不是可以忽视的问题。

## SKILL.md vs scripts：明确分工边界

既然 SKILL.md 和 scripts 各有擅长，那么如何划分它们的职责边界？我建议用一个简单原则来区分：

**SKILL.md 负责思考和分发，scripts 负责执行和实现。**

为了更清晰地展示这个分工，这里有一个详细的对比：

| 维度 | SKILL.md | scripts |
|------|----------|---------|
| **核心职责** | 思考与决策 | 执行与实现 |
| **5W1H 定位** | Why/Who/When/Where/How | What |
| **比喻** | 大脑 | 手脚 |
| **输入** | 用户意图、上下文 | 参数、文件路径 |
| **输出** | 决策、方案、指令 | 执行结果、数据 |
| **修改原因** | 业务逻辑变化 | 性能优化、bug 修复 |
| **擅长场景** | 理解意图、制定方案 | 精确执行、批量处理 |

具体来说：

- **SKILL.md**：理解用户意图（Why），分析用户是谁（Who），判断什么时候做什么（When/Where），决定怎么做（How）。它是"大脑"，负责决策和规划。
- **scripts**：执行具体动作（What）。它是"手脚"，负责干活。

```
┌─────────────────────────────────────────────────┐
│          SKILL.md 与 scripts 协作流程             │
├─────────────────────────────────────────────────┤
│                                                 │
│   用户输入                                       │
│      │                                          │
│      ▼                                          │
│   ┌─────────────────┐                           │
│   │   SKILL.md      │                           │
│   │   (大脑)        │                           │
│   │                 │                           │
│   │ • 理解意图      │                           │
│   │ • 制定方案      │                           │
│   │ • 决策分发      │                           │
│   └────────┬────────┘                           │
│            │ 调用指令                           │
│            ▼                                    │
│   ┌─────────────────┐                           │
│   │   scripts/      │                           │
│   │   (手脚)        │                           │
│   │                 │                           │
│   │ • 精确执行      │                           │
│   │ • 返回结果      │                           │
│   └────────┬────────┘                           │
│            │                                    │
│            ▼                                    │
│      返回给 SKILL.md                            │
│            │                                    │
│            ▼                                    │
│      输出给用户                                 │
│                                                 │
└─────────────────────────────────────────────────┘
```

以 file-organizer 为例。SKILL.md 的工作是：理解用户想要整理文件，询问源目录和目标目录，分析有哪些文件、应该如何分类，决定移动策略。然后它调用 scripts 中的 check-duplicates.py 来检查重复文件，调用 organize.py 来执行移动操作。scripts 不需要"思考"，它只需要按照指令完成具体的文件操作。

这个分工的好处是：职责清晰，易于维护。SKILL.md 关注"智能"，scripts 关注"能力"。当需要修改业务逻辑时，你改 SKILL.md；当需要优化执行效率时，你改 scripts。

### 实战技巧：用序号命名提升执行遵循性

当你的 Skill 需要按步骤执行复杂任务时，scripts 的命名方式会直接影响 Claude 的执行遵循性。

**问题场景：** 假设一个数据处理任务需要五个步骤：数据采集、清洗、转换、验证、输出。如果你的 scripts 文件名是 `collect.py`、`clean.py`、`transform.py`、`validate.py`、`output.py`，Claude 在执行时可能会搞混顺序或跳过某些步骤。

**解决方案：** 在文件名前加上序号前缀：

```
scripts/
├── 01-collect.py      # 第一步：数据采集
├── 02-clean.py        # 第二步：数据清洗
├── 03-transform.py    # 第三步：数据转换
├── 04-validate.py     # 第四步：数据验证
└── 05-output.py       # 第五步：结果输出
```

这个简单的命名技巧带来三个好处：

| 好处 | 说明 |
|------|------|
| **执行顺序清晰** | 序号明确定义了调用顺序，减少混乱 |
| **提示词更精确** | 可以直接写"执行 01-collect.py，然后执行 02-clean.py" |
| **遵循性提升** | Claude 会严格按照序号顺序执行，不会跳步 |

在 SKILL.md 中调用时也可以更加明确：

```markdown
## 执行流程

1. 运行 `scripts/01-collect.py` 采集原始数据
2. 运行 `scripts/02-clean.py` 清洗数据
3. 运行 `scripts/03-transform.py` 转换数据格式
4. 运行 `scripts/04-validate.py` 验证结果
5. 运行 `scripts/05-output.py` 输出最终文件
```

这种命名方式特别适合：
- 多步骤的批处理任务
- 有严格依赖关系的流程
- 需要保证执行顺序的场景

记住：**序号前缀是低成本、高回报的技巧，它能让 Skill 的执行更加可控和可预测。**

## templates 和 references：被忽视的资产

除了 scripts，templates 和 references 两个目录也经常被忽视。

**templates 目录**存放模板文件。如果你的 Skill 需要生成特定格式的内容——比如代码模板、文档模板、邮件模板等等——把这些模板放在 templates 中，执行时根据用户需求选择并填充，可以大大提高输出的一致性和效率。

| Skill 类型 | templates 用途 | 示例 |
|-----------|---------------|------|
| **代码生成** | 存放代码框架模板 | React 组件模板、API 路由模板 |
| **文档生成** | 存放文档格式模板 | README 模板、API 文档模板 |
| **邮件生成** | 存放邮件格式模板 | 商务邮件模板、通知邮件模板 |
| **报告生成** | 存放报告结构模板 | 周报模板、项目报告模板 |

这不是必需的，但对于需要格式化输出的 Skill 来说非常有用。我自己的经验是：当发现同一个格式的输出需要重复生成三次以上时，就可以考虑把它做成模板了。

### 真实案例：github-smart-commit 的 templates 实现

为了更具体地展示 templates 的价值，这里有一个来自 `github-smart-commit` 的真实案例。这个 skill 需要生成符合 Conventional Commits 规范的 Git commit 消息，为此它提供了完整的模板系统。

以下是其 `.gitmessage` 模板文件（节选自 GitHub `akira82-ai/github-smart-commit`）：

```
# <类型>(<可选范围>): <简短描述>
# |<-------- 最多 50 个字符 -------->|

# 解释为什么做这个改动（what & why，不是 how）
# |<---------------------- 尽量每行不超过 72 个字符 ---------------------->|

# 提供相关 issue、文章或其他资源的链接
# 示例：Fixes #23, Related to #456

# ─────────────────────────────────────────────────────────────
# 提交类型（type）
# ─────────────────────────────────────────────────────────────
# feat : 新功能
# fix : 修复 bug
# docs : 文档变更
# style : 代码格式调整（不影响功能）
# refactor : 重构（既不是新功能也不是修复）
# perf : 性能优化
# test : 测试相关
# build : 构建系统或依赖变更
# ci : CI 配置变更
# chore : 其他不修改 src 或 test 的变更
# revert : 回退之前的提交

# ─────────────────────────────────────────────────────────────
# 示例
# ─────────────────────────────────────────────────────────────
# feat: 添加用户认证功能
#
# 实现 JWT 认证，支持登录、注册、登出功能
# - 添加登录页面
# - 实现 token 刷新机制
# - 添加权限验证中间件
#
# Closes #123

# fix: 修复用户登录时密码验证错误的问题
#
# 密码比较时未使用 bcrypt，导致明文比较失败
# 现在使用 bcrypt.compare() 进行验证
#
# Fixes #456

# ─────────────────────────────────────────────────────────────
# 规范参考
# ─────────────────────────────────────────────────────────────
# 1. 标题不超过 50 个字符
# 2. 标题首字母大写（中文无此要求）
# 3. 标题不以句号结尾
# 4. 标题使用祈使语气（"添加"不是"添加了"）
# 5. 正文每行不超过 72 个字符
# 6. 正文解释 what 和 why，不是 how
# 7. 破坏性变更用 feat! 或 BREAKING CHANGE 标记
```

这个模板展示了 templates 的几个核心价值：

**标准化输出。** 模板定义了严格的格式规范——标题长度、正文宽度、类型分类。这确保了每次生成的 commit 消息都符合 Conventional Commits 规范，避免了格式不一致的问题。

**教育作用。** 模板不仅是填充格式，它还包含了详细的规范说明和示例。用户在使用模板的过程中，会逐渐学会如何写好 commit 消息，这是一种"边用边学"的教育方式。

**减少决策负担。** 当用户需要写 commit 消息时，不需要从头思考"应该怎么写"，只需要按照模板的结构填充内容。这大大降低了认知负担，提高了使用效率。

此外，项目还提供了 `install-template.sh` 脚本，让用户可以一键安装 Git commit 模板：

```bash
#!/bin/bash
# Git Commit 模板安装脚本
# 用于配置 Git 使用 .gitmessage 作为默认 commit 模板

set -e

SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
TEMPLATE_FILE="$SCRIPT_DIR/.gitmessage"

# 检查模板文件是否存在
if [ ! -f "$TEMPLATE_FILE" ]; then
    echo "❌ 错误：找不到模板文件 $TEMPLATE_FILE"
    exit 1
fi

# 复制到用户主目录
echo "📋 安装 Git Commit 模板..."
cp "$TEMPLATE_FILE" "$HOME/.gitmessage"

# 配置 Git 使用模板
git config --global commit.template "$HOME/.gitmessage"

echo "✅ 安装完成！"
echo ""
echo "现在每次执行 'git commit' 时，都会看到这个模板"
```

这个脚本展示了 templates 与 scripts 的协作关系：templates 提供内容模板，scripts 负责模板的分发和安装。两者配合，让 Skill 的价值得以最大化。

**references 目录**是 Skill 的知识库。这里可以存放任何你觉得有用的参考资料——行业文档、技术规范、案例集合、术语表、FAQ 等等。这些资料会在 Skill 执行时被提供给 Claude 作为参考，确保输出符合你的专业要求和风格偏好。

| 参考类型 | 用途 | 示例 |
|---------|------|------|
| **行业文档** | 提供行业标准知识 | 编码规范、设计指南 |
| **技术规范** | 确保技术准确性 | API 文档、协议说明 |
| **案例集合** | 提供实际参考 | 历史案例、最佳实践 |
| **术语表** | 统一术语使用 | 专业术语词典、缩写表 |
| **FAQ** | 处理常见问题 | 用户问答、故障排查 |

但这里有一个**核心困境**：references 的私有化问题。

如果你的 Skill 是通用型的——比如"代码格式化工具"、"翻译助手"——那么你的 references 可以是公开的，可以被任何人使用，你的 Skill 也可以被自由分享和传播。

但如果你的 Skill 包含了私有知识——比如公司内部的文档规范、个人的专业案例、特定的术语定义——这些内容不适合明文传播。你不能把包含这些知识的 Skill 发布到 GitHub 上，因为那是公司机密或个人资产。

这就导致了一个矛盾：**最有价值的 Skills，往往是最难被分享的。**

那些包含丰富专业知识、有独特案例库、有深度行业洞察的 Skills，它们的参考库是核心价值所在。但恰恰是这些内容，无法被公开分发。目前这个问题没有好的解决方案——你可以尝试加密混淆，或者搭建在线知识库，但这些方案要么效果有限，要么实现复杂。

我把它提出来，是为了让你意识到这个困境的存在。如果你的 Skill 依赖私有知识，你需要提前规划：这个 Skill 是给自己用的，还是准备分享给团队或公众？如果是后者，如何处理私有知识？这些问题需要在设计阶段就想清楚。

## 实战建议：渐进式增强

目录结构的学习和采用应该是渐进的。我建议按照以下顺序逐步掌握：

```
┌─────────────────────────────────────────────────┐
│          Skill 目录结构渐进式掌握路径             │
├─────────────────────────────────────────────────┤
│                                                 │
│   阶段 1: 只用 SKILL.md                         │
│   ┌─────────────────────────────────────┐       │
│   │ • 熟悉 prompt 编写                  │       │
│   │ • 理解 Skill 调用机制               │       │
│   │ • 完成基本功能                      │       │
│   └─────────────────────────────────────┘       │
│                    │                            │
│                    ▼ 需要更多能力时              │
│   阶段 2: 加入 references/                     │
│   ┌─────────────────────────────────────┐       │
│   │ • 积累专业知识                      │       │
│   │ • 提高输出质量                      │       │
│   │ • 形成知识资产                      │       │
│   └─────────────────────────────────────┘       │
│                    │                            │
│                    ▼ 需要格式化输出时            │
│   阶段 3: 加入 templates/                     │
│   ┌─────────────────────────────────────┐       │
│   │ • 统一输出格式                      │       │
│   │ • 提高效率                          │       │
│   │ • 保证一致性                        │       │
│   └─────────────────────────────────────┘       │
│                    │                            │
│                    ▼ 需要精确执行时              │
│   阶段 4: 加入 scripts/                        │
│   ┌─────────────────────────────────────┐       │
│   │ • 扩展执行能力                      │       │
│   │ • 提高精确度                        │       │
│   │ • 注意安全问题                      │       │
│   └─────────────────────────────────────┘       │
│                                                 │
└─────────────────────────────────────────────────┘
```

每个阶段都有其价值，不要急于跳到复杂的阶段。当你发现当前阶段的能力不够用时，自然会有动力进入下一个阶段。

## 写在最后

目录结构不是一个技术细节，而是产品思维的体现。

一个只写 SKILL.md 的 Skill，就像一个没有五脏六腑的身体——它能动，但能力有限。一个充分利用目录结构的 Skill，才是一个完整的"有机体"——有大脑（SKILL.md），有手脚，有记忆，有工具。

但这同时也意味着更大的责任。高权限带来强大能力，也带来安全风险；丰富功能带来更好体验，也带来更复杂的维护。作为开发者，你需要在这些之间找到平衡。

记住产品设计的原则：**够用就好，不要为了复杂而复杂。** 如果一个简单的 SKILL.md 就能满足需求，那就不要强行加 scripts、templates、references。但当你发现 SKILL.md 不够用时，记住：你还有很多武器可以用。
