# 第六部分：超越执行者——Skill 向 Agent 的进化

> "AutoSkill 是一个微缩版的、精简版的 Agent"
>
> "从执行者到规划者，Skill 的定位发生了根本性的转变"
>
> "动态任务拆分、智能匹配、自动安装、Task 编排——这是前无古人的尝试"

## 一个元技能的诞生

在前面五章中，我们讨论了 Skills 的本质、产品化思维、参数化设计、目录结构实践和调用体系。这些内容构建了一个完整的认知框架：理解什么是 Skill、如何用产品化的方式思考它、如何设计它的接口、如何组织它的内部结构、如何让它调用 Claude Code 的能力。

但所有这些讨论，都是围绕"单个 Skill"展开的。我们假设每个 Skill 都是独立的工具，它们各自解决一个特定问题。如果我们问：能不能有一个"管理其他 Skills 的 Skill"？能不能有一个 Skill，它自己不做任何具体工作，只负责理解和拆分用户需求，然后找到合适的 Skills 来完成？

这就是 AutoSkill 的由来。

**AutoSkill 是一个"元技能"**——它不直接完成任何任务，而是负责管理其他 Skills。它理解用户意图，拆分任务，匹配 Skills，生成 Tasks，然后退出。剩下的工作由它调度起来的 Skills 和 Tasks 去完成。

当我第一次让 AutoSkill 跑通的时候，我是真的拍桌子欢呼。那一刻我意识到：**这就是一个微缩版的、精简版的 Agent。** 它能理解、规划、匹配、编排。这不就是 Agent 的核心能力吗？

而且，我敢说：**我没有见过第二个人这么去用它的。** 这不是自夸，而是陈述一个事实。AutoSkill 代表了一种前无古人的尝试，它打开了 Skills 的新可能。

**注意**：AutoSkill 目前是一个概念验证项目和实战案例。与前面章节中展示的 `file-organizer` 和 `idea-to-post` 不同，AutoSkill 可能尚未在 GitHub 上公开发布（截至本书写作时，`akira82-ai/AutoSkill` 仓库不存在）。本章将重点讲解 AutoSkill 的设计思想和工作流程，而非具体的源代码实现。

## AutoSkill 的完整工作流程

AutoSkill 的工作流程可以拆解为以下步骤：

| 步骤 | 动作 | 说明 | 输出 |
|------|------|------|------|
| **1** | 用户输入任务 | 自然语言描述需求 | "从三个数据源采集、合并、发送" |
| **2** | 意图理解 | 可执行性优先于准确性 | 理解后的意图 |
| **3** | 任务拆分 | 拆分成可独立执行的步骤 | 子任务列表 |
| **4** | 扫描 Skills | 动态扫描已安装的 Skills | Skills 列表 |
| **5** | 匹配 Skills | 语义匹配每个子任务 | Skill-子任务映射 |
| **6** | 自动安装 | 缺失的 Skill 自动下载 | 完整的 Skills 覆盖 |
| **7** | 生成 Tasks | 为每个子任务生成 Task | Tasks 列表 |
| **8** | AutoSkill 退出 | 规划完成，退出执行 | Tasks 开始独立执行 |
| **9** | Tasks 执行 | 独立、容错、可追溯 | 最终结果 |

**第一步：用户输入任务。**

用户用自然语言描述他想要完成的事情。比如"我要从三个不同的数据源采集数据，然后在本地合并处理，最后发送到某个系统"。这个描述可能是模糊的、不完整的，但没关系。

**第二步：意图理解。**

AutoSkill 用一个简单的 prompt 来理解用户想要做什么。这里有个重要的设计决策：我不追求意图理解的"准确性"，因为意图理解极度复杂，没有任何系统能够保证完全理解用户意图。

我追求的是"可执行性"。即使意图理解有偏差，只要后续的执行是通过第三方 Skill 去做的，用户就能发现偏差并纠正。可执行性比准确性更重要，因为用户可以通过反馈来修正错误，但无法修正一个无法执行的系统。

**第三步：任务拆分。**

AutoSkill 把用户的需求拆分成具体的步骤。比如上面的例子可能被拆成：
1. 连接系统 A 采集数据
2. 在本地进行数据清洗
3. 连接系统 B 进行处理
4. 把结果发送到系统 C

这些步骤形成一个工作流，每一步都可以被独立执行。

**第四步：扫描已安装的 Skills。**

AutoSkill 会扫描 Claude Code 的专属目录，看看用户已经安装了哪些 Skills。这个列表是动态的——用户可以随时安装新的 Skills，AutoSkill 每次执行时都会重新扫描。

**第五步：为每个子任务匹配 Skill。**

这是 AutoSkill 最核心的能力之一。对于每个子任务，AutoSkill 会逐个读取已安装 Skills 的 description，进行语义匹配。比如子任务是"采集数据"，AutoSkill 会找 description 里包含"采集"、"爬取"、"抓取"、"fetch"等关键词的 Skill。

匹配的结果有三种可能：
- 刚好有一个 Skill 匹配
- 有多个 Skills 匹配
- 没有任何 Skill 匹配

**第六步：自动下载安装。**

如果某个子任务没有匹配的 Skill，AutoSkill 会尝试自动下载。这里有个重要的安全考虑：我只从 skills.sh 下载，并且通过标准的 npm 命令进行。自动下载第三方 Skill 有安全风险，所以必须限制下载来源，只从可信的渠道获取。

**第七步：生成 Task。**

当所有子任务都匹配了 Skills 之后，AutoSkill 会为每个子任务生成一个 Task，然后把这些 Tasks 塞入 Claude Code 的 Task 机制。

**第八步：AutoSkill 退出。**

这是 AutoSkill 与传统 Skill 最大的不同。传统 Skill 会一直执行到任务完成，但 AutoSkill 生成 Tasks 后就退出了。剩下的工作由 Tasks 独立执行。

**第九步：Tasks 开始独立执行。**

Tasks 按照 AutoSkill 设计的顺序逐步执行。如果某个 Task 出错了，用户可以纠正它，然后继续执行。后续的 Tasks 不受影响。这种容错能力是 Task 机制的优势，也是 AutoSkill 选择 Task 而不是自己执行的原因。

## 一个未解决的难题：多个 Skill 匹配时如何选择？

AutoSkill 的设计中有不少技术难点，但最大的难题是：当多个 Skills 都匹配用户需求时，应该选择哪个？

举个例子：用户说"写一篇文章"。AutoSkill 可能匹配到三个 Skills：
- idea-to-post：description 是"灵感到内容创作"
- article-writer：description 是"文章写作工具"
- blog-generator：description 是"博客生成器"

应该选哪个？

这个问题看似简单，但实际上非常复杂。我思考过几个可能的解决方案：

| 方案 | 说明 | 优点 | 缺点 |
|------|------|------|------|
| **相关度评分** | 计算相似度，选最高分 | 自动化、无需用户干预 | 评分算法难以设计准确 |
| **使用频率** | 优先选最常用的 Skill | 符合用户习惯 | 首次使用、频率相同时失效 |
| **让用户选择** | 用 AskUserQuestion 确认 | 最稳妥、准确 | 增加操作负担 |
| **组合使用** | 调用多个 Skills 比较 | 用户可对比 | 成本高、用户可能不愿看 |
| **频率+选择** | 优先频率，首次则询问 | 平衡自动化和准确性 | 需记录用户历史 |

**方案一：相关度评分。** 计算每个 Skill 的 description 与用户意图的相似度，选分数最高的。这个方案的难点在于如何设计评分算法——简单的关键词匹配不够准确，复杂的语义匹配又难以实现。

**方案二：使用频率。** 优先选择用户最常用的 Skill。这个方案的逻辑是：用户最常用的 Skill，往往是用户最满意的。但问题在于：首次使用时怎么办？如果两个 Skills 使用频率相同怎么办？

**方案三：让用户选择。** 用 AskUserQuestion 让用户确认。这个方案最稳妥，但增加了用户的操作负担。而且如果每次都要用户选，AutoSkill 的"自动"特性就打折扣了。

**方案四：组合使用。** 同时调用多个 Skills，让用户比较结果。这个方案的问题在于成本——调用多个 Skills 需要更多时间和资源，而且用户可能不愿意看多个结果。

我目前的倾向是：**使用频率 + AskUserQuestion。** 优先选择用户最常用的 Skill；如果是首次使用或频率相同，用 AskUserQuestion 让用户选择一次，然后记住用户的选择，下次就不再问了。

这个问题值得深入思考，因为它关系到 AutoSkill 的用户体验。如果你有更好的想法，欢迎告诉我。

## 为什么用 Task 而不是 Skill 自己执行？

一个常见的问题是：为什么 AutoSkill 要生成 Tasks 然后退出？为什么不让 AutoSkill 自己执行所有工作？

答案很简单：**复杂工作流的容错性。**

如果一个 Skill 要执行一个复杂的工作流——比如上面提到的"从三个数据源采集、处理、发送"的流程，它需要跨场景、跨业务、跨主题域。任何环节出错，整个流程可能就会失败。用户不知道哪里出错了，也不知道如何纠正。

但如果是 Tasks 执行，情况就完全不同：
- 每个 Task 是独立的，出错后用户可以纠正它，然后继续执行
- 后续的 Tasks 不受影响，可以逐步运作
- Task 的执行过程有记录，用户可以追溯每一步发生了什么

这种容错能力是 Claude Code Task 机制的优势，也是 AutoSkill 选择 Task 的原因。

更重要的是，这种设计让 AutoSkill 专注于它擅长的：理解意图、拆分任务、匹配 Skills、编排 Tasks。而把执行交给 Tasks 去做。这种分工让整个系统更加清晰、更加可靠。

## AutoSkill 的独特价值

最后，我想谈谈 AutoSkill 的独特价值——为什么说它是"前无古人的尝试"？

| 独特价值 | 传统方式 | AutoSkill 方式 | Agent 特征对应 |
|---------|---------|---------------|--------------|
| **动态任务拆分** | 预定义工作流、硬编码步骤 | 根据用户输入动态生成 | 理解与规划能力 |
| **智能 Skill 匹配** | 固定调用预定义工具 | 语义匹配、自动选择 | 工具选择能力 |
| **自动安装 Skill** | 手动安装、手动配置 | 发现缺失自动下载 | 自我扩展能力 |
| **Task 编排** | Skill 自己执行到底 | 生成后退出、Task 独立执行 | 规划与执行分离 |

**第一，动态任务拆分。** 传统的自动化工具都是预先定义好工作流的，每个步骤都是硬编码的。但 AutoSkill 是根据用户的输入动态生成工作流的。每次用户的需求不同，生成的工作流也不同。这种灵活性是前所未有的。

**第二，智能 Skill 匹配。** AutoSkill 不是简单地调用预定义的 Skills，而是根据子任务的内容，智能匹配最合适的 Skill。匹配是基于语义理解的，不是基于固定规则的。这种"理解需求 -> 找工具"的能力，是 Agent 的核心特征之一。

**第三，自动安装 Skill。** 当 AutoSkill 发现缺少某个 Skill 时，它会自动下载安装。这让 AutoSkill 具备了"自我扩展"的能力——它不仅能用已有的工具，还能获取新的工具。当然，出于安全考虑，目前只从 skills.sh 下载，但这个方向是有巨大想象空间的。

**第四，Task 编排。** AutoSkill 生成 Tasks 后就退出了，让 Tasks 独立执行。这种"生成后不管"的模式，让 AutoSkill 成为了一个纯粹的"规划者"和"编排者"，而不是"执行者"。这个定位转变是根本性的。

这四个特点加在一起，让 AutoSkill 成为了一个真正的"元技能"——它管理其他 Skills，但本身不做任何具体工作。这种设计在 Claude Skills 的生态中是第一次出现，至少我没有见过第二个人这么去用它的。

## 写在最后

AutoSkill 是一个实验性的项目。它还有很多不完善的地方——比如多个 Skill 匹配时的选择问题，比如意图理解的准确性问题，比如自动下载的安全性问题。但它的价值不在于是否完美，而在于它打开了一扇门。

这扇门背后，是 Skills 从"工具"进化为"Agent"的可能性。AutoSkill 证明了：一个 Skill 可以理解意图、拆分任务、匹配工具、编排执行。这不就是 Agent 的核心能力吗？

如果你读完这一章，开始思考如何让自己的 Skill 也具备这些能力，那么 AutoSkill 的使命就完成了。
