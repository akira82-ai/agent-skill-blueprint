## D. 术语表

| 术语 | 定义 |
|------|------|
| **Skill** | 可重复的工作流指令包，基于 prompt 的元工具架构 |
| **Plugin** | 可分享的功能打包，包含多个 Skills、agents、hooks 等 |
| **Standalone** | 独立的配置方式，存放在项目的 `.claude/` 目录中 |
| **参数化** | Skill 接受命令行参数的能力，如 `/skill --param value` |
| **隐性调用** | 通过语义匹配自动触发 Skill |
| **显性调用** | 通过斜杠命令明确调用 Skill |
| **AskUserQuestion** | Claude Code 的动态交互能力 |
| **Task 机制** | Claude Code 的任务编排机制 |
| **语义化版本** | 主版本.次版本.修订版本 (MAJOR.MINOR.PATCH) 的版本管理规范 |
| **产品化思维** | 把 Skill 当作产品而非工具的思考方式 |
| **热部署** | 修改后无需重启即可生效的能力 |
| **Live Change Detection** | 实时检测文件变化并自动应用 |

## E. 常见问题

**Q: Skill 和 Plugin 有什么区别？**

A: Skill 是轻量级的能力封装，适合个人使用和快速迭代；Plugin 是重量级的功能打包，适合团队分享和标准化部署。详见番外篇的详细对比。

**Q: 如何选择参数化格式？**

A: Linux 风格 (`--param value`) 适合多参数场景，冒号风格 (`: param value`) 适合参数较少的场景。根据 Skill 的复杂度和目标用户群体选择。

**Q: 什么时候需要用 Plugin？**

A: 当你需要整合多个功能、团队标准化配置、通过 marketplace 分发时，考虑用 Plugin。个人项目和快速实验用 Skill 就够了。

**Q: AutoSkill 是什么？**

A: AutoSkill 是一个"元技能"，它能理解用户意图、拆分任务、匹配 Skills、生成 Tasks。它展示了 Skill 从执行者到规划者的转变。

**Q: 热部署是如何工作的？**

A: 通过 `--add-dir` 添加的目录中的 Skills 支持实时变化检测，修改后自动生效。Plugins 需要手动运行 `/reload-plugins` 命令。
