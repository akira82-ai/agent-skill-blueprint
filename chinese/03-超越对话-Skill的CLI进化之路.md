# 第三部分：超越对话——Skill 的 CLI 进化之路

> "参数化让 Skill 从对话工具变成 CLI 工具，灵活性大幅提升"
>
> "固定逻辑用代码固化，可变输入用参数传递，参数化的本质是分离变与不变"
>
> "不是所有 Skill 都需要参数化，需要灵活输入的才参数化"

## 从对话到命令：参数化带来的范式转变

前两章我们讨论了 Skill 的本质和产品化思维。但无论是把 Skill 当作工具还是产品，如果你只用自然语言来调用它，你就只发挥了它一半的能力。

参数化是 Skill 进化的关键一步。它让 Skill 从"对话式交互"升级为"命令式交互"，从"灵活但不精确"变成"精确且可复用"。这个转变看似简单，但影响深远。

## 大众认知的偏差

大多数人对 Skills 的认知停留在"对话式交互"层面——你用自然语言告诉 Claude 要做什么，它理解后执行。这种模式下，Skill 就像一个封装好的 prompt，每次被调用时都通过对话来获取信息。

但这个认知忽略了一个可能性：**Skill 可以像 CLI 工具一样，通过参数来接收输入。**

CLI（命令行界面）工具是程序员最熟悉的工作方式。你不会说"帮我把这个文件复制到那个地方"，而是输入 `cp source.txt target.txt`。这种方式的优点是精确、高效、可脚本化。参数化让 Skill 也具备了这些特性。

## 参数解析的本质

首先需要澄清一个误解：参数解析不是通用的、自动的。每个 Skill 的参数格式、数量、名称、含义都是自定义的，需要开发者在 SKILL.md 中明确说明。

这听起来像是一种负担，但实际上是一种自由。你可以根据 Skill 的用途设计最合适的参数格式。如果你做一个文件整理工具，可以设计 `--source` 和 `--target` 参数；如果你做翻译工具，可以设计 `--from`、`--to` 和 `--text` 参数。完全由你决定。

## 两种参数格式

Claude Code 支持两种参数格式，你可以根据习惯选择。

| 格式 | 语法 | 示例 | 适合场景 | 优点 |
|------|------|------|---------|------|
| **Linux 风格** | `/skill-name --param value` | `/organize --source ~/Downloads --target ~/Documents` | 多参数场景 | 参数清晰明确，程序员熟悉 |
| **冒号风格** | `/skill-name: param value` | `/organize: ~/Downloads ~/Documents` | 少参数场景 | 更简洁，打字更快 |

**第一种是 Linux 风格：**

```bash
/skill-name --param value
```

示例：
```bash
/organize --source ~/Downloads --target ~/Documents
```

这种格式对熟悉 Linux 命令行的用户来说非常自然，参数名称清晰明确，适合有多个参数的场景。

**第二种是冒号风格：**

```bash
/skill-name: param value
```

示例：
```bash
/organize: ~/Downloads ~/Documents
```

这种格式更简洁，适合参数较少、或者参数顺序固定的场景。你可以根据 Skill 的复杂度和目标用户群体来选择。

## 隐性调用 vs 显性调用

参数化带来了一个重要问题：这个 Skill 应该如何被调用？

| Skill 类型 | 调用方式 | 原因 | 示例 |
|-----------|---------|------|------|
| **带参数 Skill** | 显性调用（斜杠命令） | 难以匹配参数和上下文 | `/organize --source ~/Downloads --target ~/Documents` |
| **不带参数 Skill** | 隐性调用（自动触发） | 信息可从上下文推断 | 代码检测 Skill 自动触发 |

**带参数的 Skill 适合显性调用。** 也就是说，用户需要通过斜杠命令明确指定 Skill 名称和参数。为什么？因为隐性调用（通过对话自动触发）很难正确匹配参数和上下文。当用户在对话中说"整理一下我的文件"时，系统很难知道 `~/Downloads` 应该对应 `--source` 还是 `--target`。

**不带参数的 Skill 适合隐性调用。** 如果一个 Skill 不需要参数，或者所有信息都可以从对话上下文中推断出来，那么它可以被设计成自动触发。比如一个专门检测代码 bug 的 Skill，可以在用户提交代码片段时自动被调用。

这个区分很重要。设计 Skill 时，你需要先问自己：这个 Skill 需要参数吗？如果需要，就应该预期用户通过显性的斜杠命令来调用它。

## 参数化的核心价值：分离变与不变

参数化的本质是什么？是**分离固定逻辑和可变输入**。

```
┌─────────────────────────────────────────────────┐
│              参数化的本质：分离变与不变          │
├─────────────────────────────────────────────────┤
│                                                 │
│   ┌─────────────────────────────────────┐       │
│   │         固定逻辑（业务逻辑）          │       │
│   │                                     │       │
│   │   1. 扫描源目录，看看有什么文件       │       │
│   │   2. 扫描目标目录，看看有什么子目录   │       │
│   │   3. 根据文件名和目录名，匹配文件     │       │
│   │   4. 移动文件                       │       │
│   │                                     │       │
│   │   这些逻辑不会因使用场景不同而改变 │       │
│   └─────────────────────────────────────┘       │
│                                                 │
│   ┌─────────────────────────────────────┐       │
│   │         可变输入（参数）              │       │
│   │                                     │       │
│   │   • 源目录：每次都不一样             │       │
│   │   • 目标目录：每次都不一样           │       │
│   │                                     │       │
│   │   通过参数传入，逻辑保持不变       │       │
│   └─────────────────────────────────────┘       │
│                                                 │
└─────────────────────────────────────────────────┘
```

以我自己开发的 file-organizer 为例。以下是从 file-organizer 的 SKILL.md 中节选的代码，展示了如何通过参数化实现"变与不变"的分离。这段代码来自 GitHub 仓库 `akira82-ai/file-organizer` 的最新版本：

```yaml
## 执行流程

### 1. 解析参数
检查用户是否提供了源目录和目标目录参数。

### 2. 确定源目录（如未指定）

#### 2.1. 读取源目录历史记录
```bash
python3 scripts/manage_history.py read --type source
```
- 解析 JSON 输出，获取历史记录
- 保存到变量 `source_history`

#### 2.2. 询问用户
使用 `AskUserQuestion` 询问用户：
- 如果 `source_history` 不为空，将历史记录作为选项添加（标注"最近使用"）
- 提供常用目录选项（下载文件夹、桌面等）
- 提供"输入新路径"选项

**示例选项：**
```
- "/Users/xxx/Downloads"（最近使用）
- "/Users/xxx/Desktop"（历史记录）
- "下载文件夹"（~/Downloads）
- "桌面"（~/Desktop）
- "文档"（~/Documents）
- "输入新路径"
```

### 3. 确定目标目录（如未指定）

#### 3.1. 读取目标目录历史记录
```bash
python3 scripts/manage_history.py read --type target
```
- 解析 JSON 输出，获取历史记录
- 保存到变量 `target_history`

#### 3.2. 询问用户
使用 `AskUserQuestion` 询问用户：
- 如果 `target_history` 不为空，将历史记录作为选项添加（标注"最近使用"）
- 提供常用目录选项（文档文件夹、桌面等）
- 提供"输入新路径"选项

### 4. 扫描目标目录
```bash
ls -la "$target"
```
- 检查是否已有分类目录
- 如有，记录现有分类列表供后续匹配

### 5. 扫描源文件
```bash
ls -1 "$source"
```
- 获取所有文件列表
- 跳过隐藏文件（以 `.` 开头）

### 6. AI 分析并生成分类

**如果目标目录有分类**：将文件匹配到现有分类

**如果目标目录无分类**：AI 分析所有文件名，动态创建分类
```

在这个设计中：
- **固定逻辑**：扫描、分析、匹配、移动的步骤是不变的
- **可变输入**：源目录和目标目录通过参数传入，每次可以不同

这个逻辑不会因为使用场景不同而改变。但每次使用时，源目录和目标目录是不一样的——这次可能是从下载文件夹整理到文档文件夹，下次可能是从桌面整理到项目文件夹。

如果把路径写死在 SKILL.md 里，每次使用前都要修改代码，这显然不合理。参数化让路径变成了可变输入：业务逻辑保持不变，路径通过参数传入。这就是参数化的价值所在。

## 配置模式 vs 执行模式

有些 Skill 不仅需要参数，还需要配置。这两者的区别是什么？

| 对比维度 | 参数 (Parameters) | 配置 (Configuration) |
|----------|----------------|-------------------|
| **变化频率** | 每次调用都可能变化 | 设置一次后长期使用 |
| **典型示例** | 源目录、目标目录、用户输入 | API 密钥、本地路径、使用偏好 |
| **存储位置** | 命令行传入 | config.json 文件 |
| **目的** | 提供灵活性 | 提供便利性 |

**参数是每次调用时都可能变化的值。** 比如文件整理的源目录和目标目录，每次使用可能都不一样。

**配置是相对稳定的设置。** 比如 API 密钥、默认偏好、本地路径等等，这些信息设置一次后可以长期使用。

基于这个区分，一个 Skill 可以有两种运行模式：

**配置模式**（首次调用时触发）：Skill 要求用户提供配置信息——API key、本地路径、使用偏好等等。这些信息被保存到 config.json 中，之后可以被反复使用。

**执行模式**（配置完成后）：Skill 使用已保存的配置，结合用户传入的参数，执行具体任务。

不是所有 Skill 都需要配置模式。简单的一次性工具可能不需要任何配置，但复杂的、需要连接外部服务的 Skill 往往需要。在设计时问自己：这个 Skill 有没有需要持久化存储的设置？如果有，就考虑实现配置模式。

## 配置模式的实现细节

配置模式的具体实现很简单。Skill 在执行时，先检查本地目录下有没有 config.json 文件：

- 如果有，读取配置，进入执行模式
- 如果没有，要求用户填写配置，保存后进入执行模式

这里有个设计决策需要考虑：参数验证的粒度。你应该检查配置是否"正确"吗？

我的建议是：**只检查有没有值，不检查值是否正确。**

为什么？因为"正确"的定义很复杂，而且可能因人而异。如果你要求用户输入一个 API key，你只需要确认他输入了某个值，至于这个 key 是否有效、是否有权限，让他在实际使用中去发现问题就好。过度验证反而会增加用户的负担，也增加了你的维护成本。

## 什么时候需要参数化？

最后一个问题是：如何判断一个 Skill 是否需要参数化？

| 判断标准 | 需要参数化 | 不需要参数化 |
|---------|-----------|-------------|
| **输入需求** | 需要灵活输入 | 输入固定或通过对话获取 |
| **配置需求** | 需要预配置 | 无需配置 |
| **工具类型** | 通用工具型 | 一次性任务型 |
| **交互方式** | 纯命令型 | 纯对话型 |

**需要灵活、可变的输入时考虑参数化：**

- ✅ 需要用户指定文件路径或目录
- ✅ 需要用户提供自定义文本
- ✅ 需要从多个选项中选择一个
- ✅ 需要调整某些参数来控制行为

**保持简单对话式交互即可：**

- ❌ 一次性任务
- ❌ 纯对话型，通过交流获取信息
- ❌ 简单操作，不需要用户指定任何东西

参数化是一个强大的工具，但它不是必需品。不要为了参数化而参数化。记住产品设计的原则：**够用就好，复杂度是需要代价的。**
