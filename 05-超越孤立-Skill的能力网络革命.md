>
> "从执行者到规划者，Skill 的定位发生了根本性的转变"
>

## 从孤立到互联：调用体系的诞生

前面我们讨论了 Skill 的本质、产品化、参数化和目录结构。这些都是"如何构建一个 Skill"的问题。但如果我们只关注单个 Skill，就会忽略一个更大的图景：**Skills 之间可以协作，Skills 也可以调用 Claude Code 的原生能力。**

当我第一次发现这一点时，就像打开了一扇新世界的大门。Skill 不再是一个个孤立的工具，而是一个可以互相调用、可以编排、可以组合的能力网络。这个网络的强大程度，远超单个 Skill 的简单叠加。

## 三个层次的调用能力

Claude Code 为 Skill 提供了三个层次的调用能力，每一个层次都打开了新的可能性。

| 层次 | 调用能力 | 核心价值 | 使用场景 |
|------|---------|---------|---------|
| **第一层** | AskUserQuestion | 动态交互能力 | 需要用户确认、选择的场景 |
| **第二层** | Task 机制 | 从执行者到规划者 | 复杂任务拆分和编排 |
| **第三层** | Skill 调用 Skill | 形成能力网络 | Skills 之间的协作 |

我们逐一来看。

## AskUserQuestion：从被动响应到主动交互

我第一次发现 AskUserQuestion 是在观察 Claude 自带的斜杠命令时。我发现这些命令都有一个共同点：它们会在执行过程中向用户提问，并且问题的选项是根据上下文动态生成的。这种体验非常好——用户不需要一次性提供所有信息，而是在执行过程中逐步被引导。

我当时就想：这个体验这么好，能不能让 Skill 也用？

答案是肯定的。而且当我第一次成功在 Skill 中调用 AskUserQuestion 时，那种惊喜感难以言表。我的 Skill 突然具备了主动跟用户沟通和交互的能力——它不再是被动地接收输入，而是可以在执行过程中根据需要动态地询问信息、确认意图、提供选项。

AskUserQuestion 的核心参数和能力：

| 参数 | 说明 | 示例 |
|------|------|------|
| **questions** | 问题数组，每个问题包含标题、选项等 | `[{question: "选择语言", options: [...]}]` |
| **header** | 问题标题/标签，最多12字符 | "语言选择"、"目标平台" |
| **multiSelect** | 是否允许多选 | true/false |
| **options** | 选项数组，包含label和description | `[{label: "中文", description: "简体中文"}]` |

### 真实案例：idea-to-post 的渐进式追问

这里有一个来自 GitHub 仓库 `akira82-ai/idea-to-post` 的真实案例，展示了如何使用 AskUserQuestion 实现渐进式追问（节选自 SKILL.md）：

```
### Questioning Flow (7-10 Rounds of Dialogue + Multi-Stage Search)

┌─────────────────────────────────────────────────┐
│  User inputs idea                                │
└────────────────┬────────────────────────────────┘
                 │
                 ▼
┌─────────────────────────────────────────────────┐
│  [Initial Search] Background information        │
│  collection                                     │
└────────────────┬────────────────────────────────┘
                 │
                 ▼
┌─────────────────────────────────────────────────┐
│  Rounds 1-2: Direction locking                  │
│  (mainly multiple choice)                        │
│  - Goal? Audience? Platform?                     │
└────────────────┬────────────────────────────────┘
                 │
                 ▼
┌─────────────────────────────────────────────────┐
│  Rounds 3-4: Core deep dive                     │
│  (open questions)                                │
│  - What's the core viewpoint?                    │
└────────────────┬────────────────────────────────┘
                 │
                 ▼
┌─────────────────────────────────────────────────┐
│  Rounds 5-6: Real cases (required)              │
│  - When was the most recent time?                │
│  - How did you feel at that moment?              │
└────────────────┬────────────────────────────────┘
                 │
                 ▼
┌─────────────────────────────────────────────────┐
│  Round 7: Emotional resonance (required)        │
│  - Most frustrated/surprised moment?             │
└────────────────┬────────────────────────────────┘
```

这个流程图展示了 idea-to-post 如何使用 AskUserQuestion（多选题）+ 直接对话（开放问题）的组合方式，实现 7-10 轮的渐进式追问。前三轮用多选题快速锁定方向，后面用开放问题深挖内容。

**AskUserQuestion 的核心设计原则：**

1. **Multiple Choice = 骨架**：快速锁定方向，减少用户思考负担
2. **Open Questions = 血肉**：深挖内容，获取独特观点和真实案例
3. **基于前一轮答案**：每轮问题都基于用户上一轮的回答，自然过渡

这种设计让对话既有结构感（不会跑偏），又有灵活性（能深入挖掘）。

AskUserQuestion 的核心能力有三点：

**动态生成问题。** 根据当前上下文实时生成要问的问题。比如我的 file-organizer 在执行时，会先问用户"源目录是什么"，然后根据用户的回答，再问"目标目录是什么"。问题不是预先写死的，而是根据执行流程动态生成的。

以下是从 GitHub 仓库 `akira82-ai/file-organizer` 中节选的真实代码，展示了如何使用 AskUserQuestion 来询问源目录：

```yaml
#### 2.2. 询问用户
使用 `AskUserQuestion` 询问用户：
- 如果 `source_history` 不为空，将历史记录作为选项添加（标注"最近使用"）
- 提供常用目录选项（下载文件夹、桌面等）
- 提供"输入新路径"选项

**示例选项：**
```
- "/Users/xxx/Downloads"（最近使用）
- "/Users/xxx/Desktop"（历史记录）
- "下载文件夹"（~/Downloads）
- "桌面"（~/Desktop）
- "文档"（~/Documents）
- "输入新路径"
```

### 3. 确定目标目录（如未指定）

#### 3.1. 读取目标目录历史记录
```bash
python3 scripts/manage_history.py read --type target
```

#### 3.2. 询问用户
使用 `AskUserQuestion` 询问用户：
- 如果 `target_history` 不为空，将历史记录作为选项添加（标注"最近使用"）
- 提供常用目录选项（文档文件夹、桌面等）
- 提供"输入新路径"选项
```

这个设计展示了 AskUserQuestion 的三个实用技巧：
1. **利用历史记录**：从本地文件读取用户历史选择，优先显示"最近使用"的选项
2. **提供快捷选项**：常用目录（~/Downloads、~/Desktop）作为固定选项
3. **保留灵活性**：始终提供"输入新路径"选项，让用户可以自由输入

**动态生成选项。** 选项的内容也可以根据场景调整。比如我有一个翻译 Skill，它会根据用户选择的源语言和目标语言，动态生成不同的选项。这种灵活性让交互更加自然。

**控制选项位置。** 这是一个容易被忽略但非常重要的能力。你可以通过代码控制某些选项始终出现在特定位置——比如始终把"取消"选项放在最后，或者始终把"推荐选项"放在第一位。这种控制对用户体验的影响很大。

我在自己的 Skills 中大量使用了 AskUserQuestion。idea-to-post 用它来进行多轮对话探索，逐步把一个模糊的想法发展成完整的文章；file-organizer 用它来询问源目录、目标目录，并让用户确认分类计划；翻译 Skill 用它来显示翻译进度，让用户知道还要等多久。

这些实践让我深刻体会到：**好的交互不是一次性的信息收集，而是一个动态的、逐步深入的沟通过程。**

## Task 机制：从执行者到规划者

如果说 AskUserQuestion 让 Skill 更"聪明"了，那么 Task 机制则让 Skill 更"强大"了——而且这种强大体现在一个意想不到的维度：Skill 不再是执行者，而是变成了规划者。

这个转变是怎么发生的？

起因是 Claude Code 把 todo 升级成 task 的新闻。我看到这个新闻时灵机一动：既然 task 这么强大，能不能让 Skill 去生成 task，然后让 task 自己执行？

结果真的可以。

| 对比维度 | 传统模式 | Task 模式 |
|---------|---------|----------|
| **Skill 角色** | 执行者 | 规划者 |
| **工作方式** | 自己执行所有工作 | 拆分任务后退出 |
| **稳定性** | 任一出错全盘失败 | Task 独立容错 |
| **职责** | 理解+执行+输出 | 理解+规划+编排 |
| **可扩展性** | 单一 Skill 独立运行 | 多 Task 并行协作 |

**传统模式下**，Skill 自己执行所有工作。它理解用户意图，执行任务，输出结果。一切都在 Skill 的控制下。

**新模式下**，Skill 只负责规划和编排：它理解用户意图，拆分成任务，匹配需要的 Skills，生成 task，然后——Skill 退出了。剩下的工作由 task 独立执行，Skill 不再参与。

```
┌─────────────────────────────────────────────────┐
│           Task 模式工作流程                       │
├─────────────────────────────────────────────────┤
│                                                 │
│   用户输入                                       │
│      │                                          │
│      ▼                                          │
│   ┌─────────────────┐                           │
│   │   AutoSkill     │                           │
│   │   (规划者)      │                           │
│   │                 │                           │
│   │ • 理解意图      │                           │
│   │ • 拆分任务      │                           │
│   │ • 匹配 Skills   │                           │
│   └────────┬────────┘                           │
│            │ 生成 Task                          │
│            ▼                                    │
│   ┌─────────────────────────────────────┐       │
│   │       Task 1, Task 2, Task 3...     │       │
│   │                                     │       │
│   │   • 独立执行                         │       │
│   │   • 并行运行                         │       │
│   │   • 容错恢复                         │       │
│   └────────┬────────────────────────────┘       │
│            │                                    │
│            ▼                                    │
│      返回给用户                                 │
│                                                 │
└─────────────────────────────────────────────────┘
```

这个转变看似简单，但影响深远。

**首先是稳定性的提升。** Task 是 Claude Code 的原生机制，非常稳健。相比之下，Skill 自己执行复杂工作流时，任何环节出错都可能导致整个流程失败。而 Task 机制有完善的容错能力——如果某个 task 出错了，用户可以纠正它，然后继续执行；后续的 task 不受影响，逐步运作。

**其次是职责的清晰化。** Skill 专注于它擅长的：理解意图、规划任务、编排资源。Task 专注于它擅长的：稳定执行、容错恢复、进度追踪。这种分工让整个系统更加高效。

**最后是可扩展性的增强。** 当 Skill 生成 task 后退出，task 可以独立运行、可以被监控、可以被管理。这意味着一个 Skill 可以同时触发多个 task，它们可以并行执行、可以互相依赖、可以形成复杂的任务网络。

当我第一次让 AutoSkill 成功生成并执行 task 时，我是真的拍桌子欢呼。那一刻我意识到：**AutoSkill 就是一个微缩版的、精简版的 Agent。** 它能理解意图、拆分任务、匹配工具、编排执行。这不就是 Agent 的核心能力吗？

### 真实案例：insights-zh 的 Task 子代理调用

为了更具体地展示 Task 机制的实战价值，这里有一个来自 `insights-zh` 的真实案例。这个 skill 需要翻译大量 HTML 文本块（通常 300+ 块），如果直接在主会话中调用翻译，会导致嵌套会话问题，翻译结果难以传递回主会话。

解决方案是：使用 Task 工具启动独立的子代理来执行翻译任务。以下是来自 GitHub `akira82-ai/insights-zh` 的执行流程（节选）：

```yaml
### 步骤 2: 批量翻译（显示进度）

使用批量翻译模式，在主会话中实时显示进度：

1. **启动批量翻译**：
   ```bash
   python3 ~/.claude/skills/insights-zh/scripts/auto_batch.py
   ```

2. **查看进度和待翻译内容**：
   输出示例：
   ```
   [██████░░░] 50.3% (140/309)

   待翻译批次：15 行
   ═══════════════════════════════════════════════════════════

   1. Conversation Insights
   2. Dashboard Overview
   3. Time Period
   4. Last 7 days
   5. Total conversations
   ...
   ```

3. **提供翻译**：
   - 翻译显示的 15 行内容
   - 每行一个翻译
   - 保持专有名词和技术术语不变

4. **更新 TSV 文件**：
   使用以下命令更新翻译：
   ```bash
   python3 ~/.claude/skills/insights-zh/scripts/auto_batch.py --update <start_idx> <trans1> <trans2> ...
   ```

5. **重复步骤 1-4**，直到显示：
   ```
   ✅ 所有行已翻译完成！
   ```

6. **继续到步骤 3**：合并翻译结果
```

这个设计展示了一个巧妙的架构：**主会话负责控制和显示，子会话负责执行翻译。**

主会话调用 `auto_batch.py` 脚本，脚本显示当前进度和待翻译内容。主会话提供翻译后，脚本更新 TSV 文件。这个过程循环进行，直到所有内容翻译完成。然后主会话调用另一个脚本将翻译结果合并回 HTML。

这种设计的优势是什么？

**避免嵌套会话问题。** 如果直接在主会话中调用翻译 skill，会形成嵌套会话，翻译结果难以传递。使用 Task 工具启动子代理，子代理独立执行翻译，结果直接写入 TSV 文件，主会话读取文件即可。

**进度可见可控。** 主会话始终控制翻译进度，用户可以看到"已翻译 140/309"这样的实时反馈。如果某个批次翻译不满意，可以重新翻译该批次。

**容错能力强。** 如果某个翻译子代理出错，只会影响当前批次。主会话可以检测到错误，重新启动该批次的翻译。后续已完成的批次不受影响。

## Skill 调用 Skill：隐性调用体系

第三个层次是 Skill 之间的互相调用。这是最容易被忽视，但最有趣的能力。

| 调用方式 | 触发者 | 用户体验 | 使用场景 |
|---------|-------|---------|---------|
| **显性调用** | 用户 | 明确指定 Skill 精确控制 | 用户清楚知道用什么 Skill |
| **隐性调用** | Skill | 自动匹配 Skill 无感知 | 用户不确定用哪个 Skill |

**显性调用**大家都熟悉：用户通过斜杠命令直接调用某个 Skill。比如输入 `/organize`，file-organizer 就开始执行。

**隐性调用**则不同：用户根本没有指定要调用哪个 Skill，是另一个 Skill 根据上下文自动决定调用哪个 Skill。

```
┌─────────────────────────────────────────────────┐
│         AutoSkill 隐性调用工作流程                │
├─────────────────────────────────────────────────┤
│                                                 │
│   用户输入："我要写一篇文章"                      │
│      │                                          │
│      ▼                                          │
│   ┌─────────────────┐                           │
│   │   AutoSkill     │                           │
│   │                 │                           │
│   │  1. 读取已安装的 Skills 列表                │
│   │  2. 逐个读取 description                   │
│   │  3. 语义匹配：write/写作/创作              │
│   │  4. 选择最匹配的 Skill                     │
│   └────────┬────────┘                           │
│            │ 隐性调用                           │
│            ▼                                    │
│   ┌─────────────────┐                           │
│   │  idea-to-post   │                           │
│   │                 │                           │
│   │  • 开始执行内容创作流程                     │
│   │  • 多轮对话探索想法                        │
│   │  • 生成完整文章                            │
│   └─────────────────┘                           │
│                                                 │
└─────────────────────────────────────────────────┘
```

这个机制是怎么工作的？

以 AutoSkill 为例。当用户说"我要写一篇文章"时，AutoSkill 会：
1. 读取已安装的 Skills 列表
2. 逐个读取它们的 description
3. 进行语义匹配：哪些 Skills 的 description 里包含"write"、"写作"、"创作"之类的关键词？
4. 选择最匹配的 Skill
5. 隐性调用它

整个过程对用户来说是透明的。他只是表达了一个需求："写文章"。系统自动找到最合适的 Skill 来完成这个需求。

隐性调用的价值是什么？

**首先是降低使用门槛。** 用户不需要记住有哪些 Skill，也不需要知道每个 Skill 的名字。他只需要表达自己的需求，系统自动找到合适的工具。这就像你告诉助理"帮我安排一个会议"，助理会自己选择用日历还是邮件，你不需要关心。

**其次是形成能力网络。** 当 Skills 可以互相调用时，它们就不再是孤岛，而是一个互联的网络。一个 Skill 可以作为另一个 Skill 的"子程序"，多个 Skills 可以组合成更强大的能力。这种组合性让整个生态的价值指数级增长。

**最后是实现智能推荐。** 系统可以根据用户的使用习惯、Skills 的使用频率、任务的匹配程度，智能选择最合适的 Skill。这种"你不需要问，我已经知道你要什么"的体验，才是真正的产品化。

当然，这里还有一个未解决的问题：当多个 Skills 都匹配用户需求时，应该选择哪个？

比如用户说"写一篇文章"，可能匹配到 idea-to-post（灵感到内容创作）、article-writer（文章写作工具）、blog-generator（博客生成器）。应该选哪个？

我的建议是：**优先选择用户最常用的 Skill，如果频率相同，就用 AskUserQuestion 让用户选择。** 这个问题值得深入思考，因为它关系到用户体验的细节。

## 调用体系的想象空间

当我们把这三个层次的调用能力放在一起看，会发现一个更大的图景正在浮现。

| 能力层次 | 打开的可能性 | 代表案例 |
|---------|-------------|---------|
| **AskUserQuestion** | 动态交互、逐步引导 | idea-to-post 多轮对话 |
| **Task 机制** | 任务拆分、并行执行 | AutoSkill 任务编排 |
| **Skill 调用 Skill** | 能力组合、智能推荐 | AutoSkill 智能 Skill 匹配 |

AskUserQuestion 让 Skill 具备了动态交互的能力，Task 机制让 Skill 从执行者变成了规划者，Skill 调用 Skill 让孤立的能力形成了协作生态。这三者合在一起，让 Skills 从"工具集合"进化成了"能力网络"。

```
┌─────────────────────────────────────────────────┐
│            Skills 能力网络示意图                  │
├─────────────────────────────────────────────────┤
│                                                 │
│              ┌──────────────┐                   │
│              │  AutoSkill   │                   │
│              │  (协调者)    │                   │
│              └──────┬───────┘                   │
│                     │ 调用                      │
│         ┌───────────┼───────────┐               │
│         ▼           ▼           ▼               │
│   ┌──────────┐ ┌──────────┐ ┌──────────┐        │
│   │  Skill A │ │  Skill B │ │  Skill C │        │
│   │          │ │          │ │          │        │
│   │ 用AskUser│ │ 用Task   │ │ 调用Skill│        │
│   │ Question │ │ 机制     │ │ D        │        │
│   └──────────┘ └──────────┘ └─────┬────┘        │
│                              │                  │
│                              ▼                  │
│                       ┌──────────┐              │
│                       │  Skill D │              │
│                       │          │              │
│                       │  具体执行 │              │
│                       └──────────┘              │
│                                                 │
│   每一个 Skill 都是能力网络中的一个节点           │
│   它们互相调用、互相协作、动态组合                │
│                                                 │
└─────────────────────────────────────────────────┘
```

这个网络的强大之处在于：它不是预先设计好的，而是动态组合的。用户的每一个需求，都可以触发不同的 Skills 组合，形成不同的执行路径。这种灵活性和适应性，远超任何预先设计的工作流。

而且，这个网络还在不断生长。每当你添加一个新的 Skill，你就为网络增加了一个新的能力节点。这个节点可以被其他 Skills 调用，也可以调用其他 Skills。这种组合性让整个生态的价值呈现指数级增长——这不是线性叠加，而是网络效应。

想象一下未来的可能性：当你有一个包含上百个 Skills 的能力网络时，用户只需要表达一个模糊的意图，系统就能自动匹配最合适的 Skills 组合，动态生成执行方案，然后通过 Task 机制稳定执行。这已经不是"工具"的概念了，而是一个真正的"智能助理"。

而这，才是 Claude Skills 的真正魅力所在。
